## 요청 내용

MCP 클라이언트에서 태스크를 **숨기기(archive)** 하거나 **삭제(휴지통 이동)** 할 수 없다. 로그인한 사용자의 서버 권한 범위 내에서 두 동작을 MCP 도구로 수행할 수 있게 한다.

- **AS-IS**: MCP로는 태스크 숨기기·삭제가 불가능(조회/생성/수정만 지원).
- **TO-BE**: 로그인 토큰 권한에 맞춰 숨기기·삭제를 지원. 권한 미달 시 서버 403을 한글 안내로 변환.

## 배경

### 서버(gdc-service) API 조사 결과

본체 수정 없이 기존 REST 엔드포인트를 감싼다(`../gdc/gdc-service` 조회 결과, 이 레포는 클라이언트 브리지).

| 동작 | 엔드포인트 | 권한 헬퍼 | 특징 |
|------|-----------|----------|------|
| 삭제(→휴지통) | `DELETE /api/tasks/tasks/{id}/` | `can_manage_project` (프로젝트 ADMIN / 워크스페이스 Owner·Manager) | **소프트 삭제**(`is_deleted=True`). 복구 가능. 첨부·하위 연쇄 처리 |
| 숨기기 토글 | `POST /api/tasks/tasks/{id}/archive/` | `can_manage_task_visibility` (쓰기 권한 + 공개 프로젝트 WS 멤버, Viewer 제외) | `is_archived` **토글**. 하위 일괄 처리, 고정 자동 해제. **WBS 프로젝트 불가** |

- **삭제는 되돌릴 수 있는 소프트 삭제**(휴지통 이동)다 — 서버가 `is_deleted=True`로 표시하고 복구/영구삭제 액션(`restore/`, `permanent-delete/`, `trash/`)을 별도 제공한다.
- **삭제의 하위 연쇄**: WBS 프리셋은 하위 전체 연쇄 소프트 삭제(`keep_children`로 일부 승격 제외 가능), 비WBS는 직속 하위를 최상위로 승격 후 본체만 삭제.
- **숨기기는 토글**이라 도구가 현재 `is_archived`를 먼저 확인해 원하는 상태와 다를 때만 호출하면 멱등하게 동작한다.
- **복구는 단건 + 첨부만** 되돌린다 — 연쇄 삭제된 하위는 휴지통에 남고, WBS 태스크는 복구 시 원래 부모가 아니라 **최상위로 분리(weight 해제)** 된다(`views.py:890-899`). 조상 진행률 롤업은 **삭제 시에만** 수행되고 복구 시에는 하지 않는다.
- **휴지통 목록은 자동으로 현재 프로젝트로 스코프되지 않는다** — `?project=` 미전송 시 사용자가 관리(admin/owner)하는 전 프로젝트의 휴지통이 섞여 반환된다(`views.py:846-856`). 복구/삭제 권한은 모두 `can_manage_project`(관리자 이상).
- 권한은 **토큰 기준으로 서버가 강제**하므로, MCP 도구는 별도 권한 판정을 하지 않고 403을 한글 메시지로 변환한다(`delete_task_comment`와 동일 패턴).

### 범위 결정(확정)

사용자 확인 결과 아래로 확정:

- **포함**: 숨기기(`archive_task`) · 삭제(`delete_task`, soft) · 복구(`restore_task`) · 휴지통 목록(`list_trashed_tasks`).
  - 삭제는 소프트 삭제라 삭제된 태스크는 `get_task`(retrieve)에서 404, `list_*`에서도 제외된다 — 삭제 태스크 ID를 다시 찾는 API 경로는 `GET .../trash/` 뿐이므로, 복구를 실사용하려면 휴지통 목록이 함께 필요하다.
  - **태스크 지정(제목 해석)**: 네 도구 모두 `task_id`를 `int|str`로 받아 제목 해석을 지원한다(기존 `get_task`/`edit_task_description`과 동일 UX). 단, **삭제된 태스크는 일반 search에서 제외**되므로 `restore_task`의 제목 해석은 `_resolve_task`가 아니라 **휴지통 목록(`trash/`)에서 제목 매칭**으로 처리한다.
  - **확인 게이트(숨기기·삭제·복구)**: `archive_task`·`delete_task`·`restore_task` 세 도구 모두 `confirm=False`(기본)면 **해석된 대상(#번호·제목·부작용)만 반환하고 실행하지 않는다**. `confirm=True`로 재호출해야 실제 실행 — 도구가 직접 묻는 게 아니라, 미리보기 반환 → Claude가 사용자에게 재확인 → `confirm=True` 재호출 흐름이다(`task_from_doc` 확인 게이트와 동일 패턴). 제목 오해석·예상 밖 부작용(숨기기 하위 일괄 처리, 복구 시 WBS 최상위 분리·하위 미복구)으로 인한 사고를 차단한다.
- **제외**: 영구 삭제(`permanent-delete/`) — 되돌릴 수 없어 이번 범위에서 제외.
- **제외**: 조회 도구의 `show_archived` 옵션 — 추가하지 않는다.
  - **알려진 동작(의도됨)**: 서버는 `list`/`children_batch`에서만 `is_archived=True`를 제외하고 `get_task`(retrieve)는 archived를 필터하지 않으므로, **숨긴 태스크는 `list_my_tasks`/`list_tasks` 목록에서 항상 빠지고 `get_task`로 조회하려면 id로 지정해야 한다**. `get_task`의 **제목 해석은 list 검색 기반이라 숨긴 태스크를 못 찾는다**(검증 중 확인). 조회 도구에 `show_archived`는 추가하지 않는다.
  - **예외(도구 내부)**: `archive_task`(숨김 해제)·`delete_task`는 숨긴 태스크 자체가 대상이 될 수 있어 **제목 해석 시에만 내부적으로 `show_archived=true`를 붙인다**(`_resolve_task(show_archived=True)`). 사용자에게 `show_archived` 파라미터를 노출하지는 않는다.

## 로컬 사전 검증

테스트 WS 3에서 임시 태스크로 E2E 수행(직접 로컬 코드 호출 — 실행 세션 MCP는 구버전이라 미노출). 컨텍스트는 검증 전후로 운영값(ws6/p16) 복원 확인.

- **p45(이슈관리, non-WBS)**: 임시 태스크 생성 → `archive_task` 게이트(미실행 확인)·적용·멱등(이미 숨김 시 무호출)·**제목으로 숨김 해제**(show_archived 해석) → `delete_task` 게이트(미삭제 확인)·삭제 → `list_trashed_tasks`에 `deleted_at`/`deleted_by`와 함께 노출 → `restore_task` 게이트·복구(재조회 가능) → `permanent-delete`로 완전 원복. **PASS**
- **p46(WBS)**: 임시 태스크 생성 → `archive_task`가 `wbs_archive_disabled`를 한글 오류로 차단(기대대로) → 삭제·복구 왕복 후 `weight=None`·미삭제 확인 → 완전 원복. **PASS**
- 검증 중 **버그 발견·수정**: `archive_task`의 숨김 해제를 제목으로 지정하면 숨긴 태스크가 list에서 빠져 다른 동명 태스크로 오해석됨 → `_resolve_task`에 내부 `show_archived` 옵션 추가, archive/delete가 이를 사용하도록 수정. 기존 pytest 95건 회귀 없음.

## 참고 사항

- 변경 파일(예정): `gdc_mcp/server.py`(도구 추가), 필요 시 `commands/`(슬래시 커맨드).
- `client.py`는 이미 `request("DELETE"/"POST", ...)`를 지원하므로 클라이언트 변경 불필요.
- **본체(gdc-service) 무수정** — 기존 엔드포인트만 소비.
- 삭제는 소프트 삭제라 안전하나, **영구 삭제(`permanent-delete/`)는 되돌릴 수 없으므로 이번 범위에서 제외**.
