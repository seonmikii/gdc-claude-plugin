## 요청 내용

작업 요청 문서(`docs/requests/**/*.md`)를 기반으로 GDC 태스크를 만드는 `task_from_doc` 로직을,
외부 참고 자산(UX 카피 티켓 등록 스킬 등)에서 도출한 검증된 패턴을 참고해 보완한다(요지는 아래 내재화).
**이 문서는 기획(계획 수립) 단계이며, 구현 전 승인 대기 상태다.**

## 배경

### 현재 흐름 (as-is)

`/gdc-task-from-doc` 커맨드/프롬프트 → 에이전트가 description 작성 → `task_from_doc` 도구 호출.

`task_from_doc` ([server.py:931](../../../gdc_mcp/server.py#L931)) 이 하는 일:

1. frontmatter에 `task_id`가 이미 있으면 재생성 안 함 → `already_linked` 반환 (**멱등성 OK**)
2. `_strip_meta_steps`로 `[작업 내용]` 블렛 중 프로세스 메타 단계(빌드/검증/테스트/커밋/INDEX 등) 제거
3. 첫 `# ` 헤딩에서 제목 추출
4. 레포 컨텍스트에서 project 자동 결정
5. 메타표 `상태` 매핑: `done → 완료`, `partial → 진행` (status 인자 없을 때만)
6. 완료 계열이면 `progress=100` + 실제 종료일=오늘 자동 주입
7. 태스크 생성 후 frontmatter에 `task_id`/`task_url` 기록

에이전트가 넘기는 description 템플릿(현재):

```
<문서 "요청 내용" 한 줄 요약>

[작업 내용]
- 작업 결과 단계 요약 1
- 작업 결과 단계 요약 2
```

### 참고한 패턴 (내재화 — 외부 SKILL 자산에서 도출, 원본은 삭제 예정)

착수 시 외부 참고 자산(UX 카피 티켓 등록 스킬 등)에서 아래 패턴을 도출해 반영했다. 원본 파일은 이 레포에서
삭제 예정이므로 요지를 여기 내재화한다:

- **description을 HTML 리치텍스트로 작성** — GDC `description`은 리치텍스트(HTML)로 저장·렌더링되며,
  평문 `\n`/`-`는 줄바꿈이 무시돼 한 줄로 붙거나 하이픈이 글자 그대로 보인다 → `<p>`(문단) + `<ul><li><p>`(불렛).
  (본 문서 Gap 1에서 실제 태스크로 **실증 확인 완료**.)
- **미리보기 + 단일 확인 게이트** — 생성 전 표로 미리보기 후 명시적 확인 1회.
- **enum 실시간 조회 + 한글 label 노출 / `name` 전달**.
- **상위 태스크 연결** — 생성 태스크를 `update_task(parent=)`로 상위에 묶음(구 Gap 4, 후속 과제).
- **구조화 표로 검토** — 생성 전 사람이 검토 가능한 형태로 정리.

## 개선 방향 (gap 분석)

현재 `task_from_doc`에 이미 반영된 것(멱등성·메타단계 제거·완료 보정·담당자 자동)은 유지한다.
위 참고 패턴을 gap으로 정리하되, **이번 범위는 Gap 1·2 두 건**이다 —
Gap 3은 Gap 2(항상 켜지는 미리보기)에 흡수되고, Gap 4는 후속 과제로 분리한다.

### ⭐ Gap 1 — description 구조·리치텍스트 개선 (권장: 필수)

두 가지를 함께 개선한다 — (1) 템플릿 구조를 라벨 섹션으로 디벨롭, (2) 이를 도구가 HTML로 변환.

- **문제 (a) 렌더링**: `task_from_doc`은 평문 마크다운(`-` 블렛 + `\n`)을 그대로 `description`으로 넘긴다.
  GDC는 이 필드를 **리치텍스트(HTML)로 저장·렌더링**하므로, 평문을 보내면 본문이 뭉개진다.
  **(실증 확인 完 — 2026-07-21, `get_task`)** 실제 태스크 description이 HTML로 저장됨을 확인:
  - #292(14957): `<ul><li><p>26/06/23 개발 서버 연동</p></li>…</ul><p></p><p><a href="…">…</a></p>`
  - #273(14847): `<ul><li><p>MCP 서버 혹은 CLI 연동을…</p></li></ul><p></p>`
  - 규칙: 블렛=`<ul><li><p>`, 문단=`<p>`, **블록 간격=빈 문단 `<p></p>`**(GDC 에디터 산출 형식).
  → 평문 전송은 비정상이고 **HTML 변환이 네이티브 형식에 맞는 방향임이 데이터로 확정.**
- **문제 (b) 구조**: 현재 템플릿은 `요약 + [작업 내용]`뿐이라 **"무엇을 어떻게 바꾸는지(전/후)"** 가
  본문에 드러나지 않는다. ux-ticket의 AS-IS/TO-BE 패턴을 태스크 본문에도 도입한다.

- **개선안 (구조) — 라벨 섹션 템플릿:**

  ```text
  [요약]
  한두 줄 요약

  [AS-IS]        ← 선택: '구현 전 상황'이 있을 때만, TO-BE와 짝으로
  구현 전 상황

  [TO-BE]        ← 선택
  구현 후 상황

  [작업 내용]
  - 산출물 단계 1
  - 산출물 단계 2
  ```
  - `[요약]`·`[작업 내용]` = **필수**, `[AS-IS]`/`[TO-BE]` = **선택(짝)**.
    작업 요청 문서에는 AS-IS/TO-BE 전용 필드가 없으므로, 에이전트가 `요청 내용`·`배경`에서 전/후 상황이
    **실제로 드러날 때만** 도출해 넣는다. 불명확하면 생략(추측·빈말 금지) — 신규 기능·문서 작업엔 보통 빠진다.
  - `[작업 내용]` 블렛의 프로세스 메타 단계 제거(`_strip_meta_steps`)·체크박스 금지는 현행 유지.

- **개선안 (렌더링) — 도구가 위 템플릿을 HTML로 변환:**
  - `[라벨]` 줄 → 볼드 라벨 문단 (`<p><strong>요약</strong></p>` 등 — `<strong>` 지원 실증 확인)
  - 라벨 아래 문단 텍스트 → `<p>...</p>`
  - `[작업 내용]`의 `-` 블렛 → `<ul><li><p>...</p></li></ul>` (연속 블렛은 하나의 `<ul>`로 묶음)
  - 블록(라벨 섹션) 사이 간격은 **빈 문단 `<p></p>`**로 넣는다 — GDC 에디터 네이티브 형식과 일치(실증 확인).
  - **텍스트 내용은 태그로 감싸기 전에 `&`·`<`·`>`를 이스케이프**한다. 작업 요청 문서 본문에는
    `progress<100`·`<p>`·`A & B` 같은 부등호/코드가 흔하므로, 미이스케이프 시 본문이 깨지거나
    의도치 않은 태그가 주입된다.
  - 변환 예시:

    ```html
    <p>요약</p>
    <p>문서 요청 내용 요약</p>
    <p>AS-IS</p>
    <p>평문 마크다운을 그대로 전송해 본문이 한 줄로 뭉개짐</p>
    <p>TO-BE</p>
    <p>도구가 HTML로 변환해 문단·불렛이 제대로 렌더링됨</p>
    <p>작업 내용</p>
    <ul><li><p>변환 헬퍼 추가</p></li><li><p>task_from_doc 연결</p></li></ul>
    ```
  - **확인 完 (2026-07-21, 태스크 #15383)**: 라벨 강조(`<strong>`) **지원됨** → 라벨을 볼드로 확정.
    섹션 간격(`<p></p>`)·블렛·이스케이프(`progress<100 & a>b` 그대로)도 UI에서 정상 렌더링 확인.

- **결정 A (확정): A-1 — 도구(`task_from_doc`) 내부에서 마크다운→HTML 변환.** 에이전트는 위 라벨
  템플릿(단순 텍스트)을 넘기고, 도구가 리치텍스트로 렌더링한다. "입력 검증·정규화는 도구 레벨" 원칙에
  부합하고, 이미 존재하는 `_strip_meta_steps` 파이프라인 뒤에 변환 단계를 이어 붙이면 된다.
  - (반려) A-2: 에이전트가 HTML 직접 작성(ux-ticket 방식) — 에이전트별 편차 위험으로 미채택.

### Gap 2 — 미리보기 + 확인 게이트 부재 (권장: 커맨드 레벨 반영)

- **문제**: `/gdc-task-from-doc`은 곧바로 태스크를 생성한다. ux-ticket은 생성 전 미리보기 + 단일 확인.
- **개선안**: 커맨드 절차에 "생성 전 미리보기(제목 / 렌더링될 본문 / 상태 / 유형 / **우선순위**)를 보여주고
  **한 번 확인받은 뒤** `task_from_doc` 호출" 단계를 추가. (도구는 그대로, 프롬프트/커맨드만 보강)
  우선순위를 포함해 확정될 값 전부를 이 미리보기 한 곳에서 보여주므로, 사용자는 원하는 항목만 그 자리에서
  변경 요청할 수 있다(Gap 3 참조).
- **결정 B (확정): 항상 확인.** 기본 동작을 "생성 전 미리보기 + 단일 확인"으로 고정한다(현행처럼 바로
  생성하지 않는다).

### Gap 3 — priority(우선순위): Gap 2 미리보기에 흡수 (별도 질문 없음)

- **문제**: 현재 status는 메타표에서 자동 매핑, task_type은 에이전트가 매칭하지만 **priority는 항상 생략**.
- **결론**: 별도 선택 질문을 추가하지 않는다. Gap 2에서 미리보기가 **항상** 켜지고 우선순위도 함께
  노출되므로, 사용자가 원하면 그 자리에서 변경 요청하면 된다. 문서 메타표 힌트 자동 매핑도 하지 않는다
  (과설계 지양 — AGENTS 규칙). → **Gap 2로 흡수, 이번 범위에서 독립 항목 아님.**

### Gap 4 — 상위/하위/연관 태스크 연결: 후속 과제로 분리

- **문제**: 문서로 만든 태스크를 다른 태스크(상위/하위/연관)와 묶을 수단이 없다.
- **결론**: 이번 범위에서 **제외**한다. 상위(parent)·하위(child)·연관(related) 태스크 연결을
  **한꺼번에 다루는 별도 후속 과제**로 넘긴다(부분 도입 시 관계 모델이 갈라질 수 있어 묶어서 설계).

## 제안 범위 (승인 시)

**단계적 확대 — task_from_doc로 먼저 구현·검증 후 전체 경로로 통일한다.**

- **1단계 (이번 범위, 파일럿)**: `task_from_doc`에 Gap 1(라벨 템플릿 + HTML 변환) + Gap 2(항상 켜지는
  미리보기·단일 확인, 우선순위 포함)를 적용한다. 실제 태스크 1건으로 렌더링·라벨 볼드까지 검증.
- **2단계 (검증 후 통일)**: 파일럿이 제대로 동작하면, **HTML 변환은 공통 레이어로 전체 태스크 생성
  경로**(`create_task`/`update_task`)에, **라벨 템플릿도 다른 생성 경로**(`gdc_task_new` 등)에
  확대 적용한다. 세부 설계·결정은 별도 [2단계 문서](20260721-112819-unify-html-description.md)로 분리했다.
- **후속 분리(별개)**: Gap 4(상위/하위/연관 태스크 연결 일괄 개선).
