## 요청 내용

작업 요청 문서의 Phase/체크박스 진척으로 자동 계산되는 진행률을, 연결된 태스크에 동기화할 때 **10% 단위로만** 반영한다.

- **AS-IS**: 문서 진행률에 따라 태스크 진행률이 자동 계산·업데이트되는데, 계산 결과가 10% 단위가 아니면(예: 33%, 67%) GDC UI에서 해당 태스크의 진행률을 수정하려 할 때 오류가 발생한다.
- **TO-BE**: 진행률을 태스크에 반영할 때 **가장 가까운 10% 단위로 반올림**하여 항상 0/10/20/…/100 값만 전송한다.

## 배경 (원인 분석)

관련 코드 검토 결과, 진행률은 다음 경로로 산출·전송된다.

1. [doc_utils.py:238](gdc_mcp/doc_utils.py#L238) · [doc_utils.py:250](gdc_mcp/doc_utils.py#L250) — `compute_phase_progress`가 `round(완료/전체 × 100)`으로 계산한다. 이 값은 Phase/체크박스 개수에 따라 임의의 정수(예: 3개 중 1개 → 33)가 나올 수 있어 **10의 배수가 아니다.**
2. [server.py:852-853](gdc_mcp/server.py#L852-L853) `sync_doc_progress`(MCP 도구) → [server.py:791](gdc_mcp/server.py#L791) `_apply_progress_sync`가 그 값을 그대로 `{"progress": new_progress}`로 PATCH 한다.
3. [server.py:1363-1364](gdc_mcp/server.py#L1363-L1364) `_cli_sync_doc`(PostToolUse 훅 경로)도 동일하게 `compute_phase_progress` → `_apply_progress_sync`를 탄다.

즉 문서 기반 동기화(도구·훅)로 들어온 비(非)10단위 값이 서버에 저장되고, 이후 UI 수정 시 서버/화면의 10% 단위 제약과 충돌해 오류가 난다.

**확정 설계**:

1. **반올림 위치 = `_apply_progress_sync` 경계.** 두 진입 경로(도구·훅)가 모두 이 함수를 공유하므로, 한 곳에서 반올림하면 서버로 나가는 진행률이 항상 10단위로 보장된다. 순수 헬퍼 `_round_progress(raw)`로 분리해 pytest로 검증한다.
2. **반올림 방식 = 문서대로 `round(p / 10) * 10`.** Python `round`는 은행가 반올림(짝수 반올림)이라 5로 끝나는 값은 짝수 쪽으로 붙는다(5→0, 25→20, 45→40, 85→80). 사용자 확인 결과 이 동작을 그대로 채택한다.
3. **상단 경계 조기완료 방지 = raw가 진짜 100일 때만 100·완료 전이.** `round(95/10)*10 == 100`이라 95~99%가 100으로 올라가면 완료+실제 종료일이 조기에 찍힌다. 이를 막기 위해 `_round_progress`는 **`raw >= 100`일 때만 100을 반환**하고, 그 미만은 `min(round(raw/10)*10, 90)`으로 캡핑한다(95%→90%). 완료 전이 판단도 반올림값이 아닌 **raw >= 100** 기준.
4. **상태/실제시작일 전이는 반올림 전 raw 값으로 판단.** 진행률 숫자는 반올림해 전송하되, "일이 시작됐는가"(진행 상태 전이·실제 시작일 설정)는 **raw > 0** 기준으로 결정한다. 이렇게 하면 5% 미만 진척(예: 21개 중 1개=4.7%)이 0%로 반올림돼도 상태는 **진행**으로 전이되고 실제 시작일이 찍힌다.

> 현재 코드의 상태 전이는 앞으로만(planned→in_progress→done) 이동하고 되돌리지(downgrade) 않으므로, 이미 진행/완료인 태스크가 0%로 되돌아가는 문제는 없다. 관건은 **첫 전이**가 반올림 0 때문에 누락되지 않게 하는 것이다.

## 참고 사항

- **반올림값과 상태 전이 분리(핵심)**: `round(p/10)*10`은 5% 미만 진척(예: 4%)을 0%로 만든다. 진행률 숫자는 0%로 보내되, [server.py:818](gdc_mcp/server.py#L818)·[server.py:827](gdc_mcp/server.py#L827)의 진행 전이·실제 시작일 판단은 **반올림 전 raw 값 기준**으로 하여 "진행률 0% + 상태 진행"이 되도록 한다. (진행률 5% 미만이어도 진행 상태로 두는 것이 요구자의 선호.)
- **테스트 정책**: 반올림은 순수 로직이므로 pytest로 검증 가능. 서버 PATCH 연동은 로컬 사전 검증(수동)으로 확인한다.
